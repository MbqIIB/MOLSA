Conversion Issues

Introduction

Before converting FrameMaker documents to DocBook format, they must be checked, and probably edited, to ensure the success of the conversion process. If the documents are well prepared, the conversion process will be smoother and the DocBook output will require less modification.

The Cúram Template and Formats

Only documents that use the standard Cúram FrameMaker template can be converted. The conversion process relies on the identification of the standard paragraph and character formats defined in the template to determine the structure and meaning of the text in the document. If these formats are used incorrectly, ambiguously, or in ways that deviate from their prescribed usage, the conversion process may fail.

Manual overrides of the predefined formats will not be preserved. For example, if the character format “Emphasis” is used, it will be converted to the DocBook emphasis element, but if an override such as the direct application of an italic font was used, it will be lost. There is no way the conversion process can determine why italics was used; it may well not have been used for emphasis, so such an assumption would be invalid.

All manual overrides will be flagged in the output DocBook document. They can be identified within a FrameMaker by placing the cursor in the paragraph or character sequence being checked and looking at the format name in the status bar in the lower left of the FrameMaker window. The paragraph format name and, if applicable, the character format name will be displayed. If either format name has an asterisk character appended, this indicates a manual override. On some occasions (typically when a character format is applied to all of the text in a paragraph), it may be impossible to eliminate the flagging of a manual override on a paragraph format. These can be ignored.

To remove manual overrides on character sequences, select the text and set the character format to “Default ¶ Font” to clear the overrides before applying the correct character formats.

Cross-References

The conversion process attempts to identify and preserve all cross-references in the document, however, there are some limitations.

Chapter cross-references may be broken due to an anomaly in the use of the Cúram template whereby a cross-reference marker is created on a master page instead of a body page. These will need to be fixed manually before conversion.

Only some of the standard cross-reference formats (and even less of the non-standard ones) are supported. One problem is the use of cross-references that target another file. Many can be resolved automatically, but there are restrictions. The table below describes the supported cross-reference formats and when they should be used. It is likely that most of the cross-references in your document are compliant, but some may need to be changed. Some formats have several names (some are just misspellings); the supported alternatives are listed.

Format Name

Description

Chapter Reference

A reference to a “ChapterNumber” paragraph. It should be the only format used for references to chapters. Any type of reference to a “ChapterTitle” paragraph is not supported.

The alternative names “ChapterReference”, “Chapter Num”, and “Chapter Ref” are supported.

Chapter & Section Reference

A reference to a “Section” or “Subsection” paragraph within a chapter. This is the preferred format for use when sections are located in a different chapter or file.

No alternatives are supported.

Appendix Reference

A reference to an “Appendix Number” paragraph. It should be the only format used for references to other appendices.

The alternative “Appendix Ref” is supported.

Appendix & Section Reference

A reference to a “Section” or “Subsection” paragraph within an appendix. This format was not included in the standard template, however, for a cross-reference to a section in an appendix from a location outside that appendix, this format should be applied or the cross-reference may not be resolved. You can import it from the conversion template described below. An ordinary section reference may work, but the probability is lower.

The alternative “Appendix Section Reference” is supported.

Section Reference

A reference to a “Section” or “Subsection” paragraph within the same chapter or appendix. If this is used to reference a section outside of the current chapter or appendix, it is less likely to be resolved correctly.

Many variations on the format name are acceptable: “Heading & Section”, “Section & Heading”, “SectionAndTitleReference”, “Subsection Reference”, and “SubSection Reference”.

Subsubection Reference

The “Subsubsection Reference” format is usually defined differently from the “Section Reference”, as this section level is not numbered, however, it will be treated like a standard section reference during conversion and is acceptable once the reference is within the same file. Again, when the reference is to another file, the chance of it working becomes slimmer.

Long, made-up words seem to be hard to spell, so the following variations have been identified and are acceptable: “Subsubection Reference” and “Subsubsection”.

Table Reference

A reference to a “TableTitle” paragraph within the same chapter or appendix. References to tables, even in the same chapter or appendix, cannot always be resolved due to limitations in the conversion facilities provided by FrameMaker.

No alternatives are supported.

Figure Reference

A reference to a “FigureTitle” paragraph within the same chapter or appendix. References to figures in other chapters or appendices may not be resolved.

No alternatives are supported.

Listing Reference

A reference to a “ListingStart” paragraph within the same chapter or appendix. References to listings in other chapters or appendices may not be resolved.

No alternatives are supported.

None of the cross-reference formats that include page numbers are supported. These should be located and changed to one of the supported formats.

References within the same file are always resolved accurately for all but tables. References to other files (and local tables) are resolved by matching the title text stored in the reference to the title text of possible targets. If there are several targets with the same title text (e.g., “Introduction”, etc.), the references cannot be resolved. In this case, a warning is issued and the reference will have to be fixed by hand later. The special formats “Chapter & Section Reference” and “Appendix & Section Reference” improve the odds by narrowing the search for titles to the target chapter or appendix respectively.

When a cross-reference cannot be resolved due to ambiguities, DocMaker will replace the reference with a marker that can be easily identified. The format of an unresolved cross-reference marker is __dm_xref_type__target@@@, where type is the type of reference, for example, table, or chapter_section, and target is some information about the target such as the title text or paragraph number. Simply find these markers as fix them when the conversion process is complete.

Vocabulary Limitations

The template used for Cúram documents does not have the rich vocabulary of formats (or tags) that are available in DocBook. The resulting DocBook output will, therefore, need to be edited to “upgrade” the document to use the new vocabulary.

The “GUIElement” character format is used for all buttons, menus, labels, etc., but DocBook distinguishes between these objects. The conversion process applies the interface element to all of them, but you should then modify your document and apply the guibutton, guimenu, guilabel, etc. elements. For menus, this can include the use of the menuchoice element that allows the effective definition of complex menu choices that include menus, sub-menus, and menu items.

The “CodeElement” character format is also widely used for many different objects such as class names, literal text, command names, source code fragments, etc. These are all marked as literal elements in the converted output, but, again, classname, command, sgmltag, etc. should be used instead and must be applied manually.

Other Issues

FrameMaker documents include a dictionary of words that are allowed in the document. When converted, these words are lost. The DocBook document will need to be spell-checked and the allowable words added to the wordlist. As it is possible for the document dictionary to contain erroneous words that were added by accident, this presents an opportunity to review the spelling within all the documents.

Quotation marks are used directly in FrameMaker documents, but the quote element is used in DocBook instead. As an author types a quotation mark in FrameMaker, it is converted to a “smart quote” by the application. These marks can cause problems when converting to different encodings and are very difficult to maintain when editing a DocBook document in a basic editor. The conversion process, however, cannot replace these quotation marks with the quote element with any confidence, so the exercise should be performed manually after conversion. Where the double quotation marks appear within source code listings, they should be “straight double quotes”[4] to comply with the syntax of that language and should not be marked with the quote element. If the template changes the style of the quotation mark used when rendering the quote element, you do not want it to affect any source code examples.

Character, such as the “smart” quotation marks mentioned above, that are not within the ISO “Latin 1” character set, appear as numeric character entities. For example, an em-dash appears as &#8212;. These can be difficult to decipher, so you can convert these into named character entities by editing the file using a suitable XML editor such as XXE and then saving it. In the case of the em-dash, it will now appear as &mdash;. This procedure is not mandatory, and while it makes the file more readable, the biggest culprits are the “smart” quotes that should be replaced with a quote element instead of character entities.

DocBook has no inherent concept of formatting beyond the expectation that whitespace in the content of certain elements (such as programlisting elements) should be preserved during rendering. This means that there is no way to specify an indented paragraph, something commonly used in Cúram documents by applying the “Indent” paragraph format. DocMaker will indent paragraphs as appropriate when rendering DocBook documents, but there is no one element in DocBook that can be applied to all indented paragraphs without the potential for identifying content incorrectly. For example, paragraphs within the same listitem element are indented to the same level by DocMaker, something that has to be manually arranged using different formats in FrameMaker. However, isolated indented paragraphs may be intended as short code samples, ad hoc lists, user input, etc. and these require the application of very different DocBook elements. These can only be resolved manually.
